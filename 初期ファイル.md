なので、ここではアプリ名を仮に **MicBar** として、`requirements.md` と `tech-stack.md` に分けてそのまま使える形で出す。

# `requirements.md`

# MicBar 要件定義書

## 1. プロダクト概要

**MicBar** は、macOSのメニューバーから入力マイクを素早く切り替えるための常駐アプリである。

このアプリは、macOS標準の「システム設定 → サウンド → 入力」へ移動せずに、メニューバー上のアイコンをクリックするだけで、接続中の入力マイク一覧を表示し、選択したマイクをシステムのデフォルト入力デバイスに設定できることを目的とする。

## 2. 開発目的

現在の課題は、入力マイクの切り替えが深い設定画面にあり、頻繁に切り替えるユーザーにとって手間が大きいことである。

MicBarでは、以下を解決する。

* ショートカットキーを増やさずにマイクを切り替えたい
* 今どのマイクが選ばれているかを視覚的に確認したい
* 配信、録音、Web会議、音声入力の前に素早く入力元を変更したい
* 出力デバイスではなく、入力マイクだけを扱いたい

## 3. 対象ユーザー

主な対象ユーザーは以下。

* Mac mini / MacBook / Studio Display / USBマイク / オーディオインターフェース / AirPodsなど、複数の入力デバイスを使う人
* Zoom、Google Meet、OBS、Discord、音声入力、録音アプリを頻繁に使う人
* キーボードショートカットを増やしたくない人
* macOS標準のサウンド設定画面を開くのが面倒な人

## 4. 基本方針

MicBarは、**入力マイク切り替え専用アプリ** とする。

出力スピーカーの切り替え、音量調整、録音、音声解析、ノイズ除去、AI文字起こしなどは初期バージョンでは扱わない。

## 5. スコープ

### 5.1 対象範囲

初期バージョンで実装するもの。

* メニューバー常駐
* 入力デバイス一覧の表示
* 現在の入力デバイスの表示
* 現在選択中の入力マイクにチェックマーク表示
* 入力マイクの切り替え
* 入力デバイスの再読み込み
* デバイス切り替え失敗時のエラー表示
* アプリ終了メニュー
* ログイン時起動の設定

### 5.2 対象外

初期バージョンでは実装しないもの。

* 出力デバイス切り替え
* システム効果音デバイス切り替え
* マイク音量調整
* マイクミュート切り替え
* 入力レベルメーター
* 音声録音
* ノイズキャンセル
* アプリごとのマイク自動切り替え
* OBS / Zoom / Discordなど外部アプリの内部設定変更
* 複数ショートカットキー管理
* iCloud同期
* チーム共有
* App Store配布

## 6. ユーザーストーリー

### US-001: メニューバーからマイクを切り替える

ユーザーとして、メニューバーのアイコンをクリックし、表示された入力マイク一覧から使いたいマイクを選びたい。
それにより、システム設定を開かずにマイクを切り替えられる。

### US-002: 現在のマイクを確認する

ユーザーとして、現在どの入力マイクが選ばれているかをメニュー内で確認したい。
それにより、Web会議や録音前のミスを減らせる。

### US-003: デバイスの抜き差しに対応する

ユーザーとして、USBマイクやAirPodsの接続状態が変わったとき、アプリ側の一覧も更新されてほしい。
それにより、古いデバイス名を選んで失敗することを避けられる。

### US-004: ショートカットを覚えずに操作する

ユーザーとして、新しいキーボードショートカットを覚えず、見たままクリックで操作したい。
それにより、既存のショートカット運用を邪魔せずに使える。

## 7. 機能要件

## F-001: メニューバー常駐

アプリは起動後、Dockに通常アプリとして出るのではなく、メニューバー上に常駐する。

メニューバーアイコンは、マイクを連想できるシンプルなアイコンとする。

## F-002: メニュー表示

メニューバーアイコンをクリックすると、入力マイクの一覧を表示する。

表示例。

```text
MicBar

現在の入力:
USB Audio

────────────

✓ USB Audio
  DELL S2725DC
  AirPods Pro
  Display Audio
  Speaker Audio Recorder

────────────

入力デバイスを再読み込み
ログイン時に起動: オフ
終了
```

## F-003: 入力デバイス一覧取得

アプリは、macOSに認識されているオーディオデバイスのうち、入力チャンネルを持つデバイスのみを一覧に表示する。

表示対象は「入力マイク」として使えるデバイスのみとし、スピーカーや出力専用デバイスは表示しない。

## F-004: 現在の入力デバイス取得

アプリは、現在macOSのデフォルト入力デバイスとして設定されているマイクを取得する。

現在の入力デバイスは、メニュー内で以下のように表示する。

* 「現在の入力: デバイス名」
* デバイス一覧内のチェックマーク
* 必要に応じてメニューバーアイコンのツールチップ

## F-005: 入力マイク切り替え

ユーザーがメニュー内の入力デバイスをクリックすると、アプリはそのデバイスをmacOSのデフォルト入力デバイスに設定する。

切り替え成功後、以下を更新する。

* チェックマーク
* 現在の入力表示
* 内部状態
* 必要に応じて短い成功表示

## F-006: デバイス再読み込み

メニュー内に「入力デバイスを再読み込み」を配置する。

ユーザーがクリックすると、接続中の入力デバイスを再取得し、メニュー内容を更新する。

## F-007: デバイス変更監視

USBマイク、ディスプレイ、オーディオインターフェース、Bluetoothデバイスなどの接続状態が変わった場合、アプリは可能な範囲で自動的に一覧を更新する。

自動検知が失敗した場合でも、「入力デバイスを再読み込み」で手動更新できること。

## F-008: エラー表示

切り替えに失敗した場合、メニュー内または軽量な通知でエラーを表示する。

想定エラー。

* デバイスがすでに切断されている
* デバイスIDが無効
* macOS側で切り替えが拒否された
* Core Audio API呼び出しが失敗した
* 現在のデフォルト入力デバイスを取得できない

エラー文言は技術者向けすぎない表現にする。

例。

```text
このマイクに切り替えられませんでした。接続状態を確認してください。
```

## F-009: ログイン時起動

ユーザーは、アプリをMacログイン時に自動起動するかどうか選択できる。

初期状態はオフでもよいが、日常利用を考えるとオンにしやすい場所へ配置する。

## F-010: 終了

メニュー内に「終了」を配置する。

Dockに表示しない設計の場合、終了導線を必ずメニュー内に用意する。

## F-011: 通知

初期バージョンでは、通知は必須ではない。

理由は、通知許可ダイアログが増えると小さなユーティリティとしての軽さが落ちるため。

ただし、将来的に以下の通知は追加可能とする。

* 「USB Audioに切り替えました」
* 「選択中のマイクが切断されました」
* 「入力デバイスを取得できませんでした」

## F-012: ショートカットキー

初期バージョンでは、グローバルショートカットキーは実装しない。

理由は、今回の主目的が「ショートカットを増やさない」ことであるため。

将来的に実装する場合も、複数ショートカットではなく、以下の1つだけを検討する。

```text
次の入力マイクに切り替え
```

## 8. 非機能要件

## NF-001: 操作性

主要操作は、メニューバーアイコンをクリックしてから2クリック以内で完了する。

理想操作。

```text
メニューバーアイコンをクリック
↓
使いたいマイクをクリック
```

## NF-002: 軽量性

常駐アプリであるため、CPU使用率とメモリ使用量は低く保つ。

常時ポーリングは避け、可能な限りCore Audioのプロパティ変更通知を使う。

## NF-003: 起動速度

ログイン時に起動する可能性があるため、起動後すぐにメニューバーアイコンを表示する。

入力デバイス一覧の取得に時間がかかる場合でも、UIをブロックしない。

## NF-004: 安定性

デバイスの抜き差し、Bluetooth接続解除、スリープ復帰後もクラッシュしない。

デバイスIDが変わる可能性を考慮し、内部的にはデバイス名だけでなくUIDも扱う。

## NF-005: プライバシー

アプリは音声を録音しない。

アプリはマイク入力ストリームを開かない。

アプリは音声データを保存しない。

アプリは外部サーバーへデータ送信しない。

アプリは分析用テレメトリを送信しない。

## NF-006: 権限

初期バージョンでは、マイク権限を要求しない設計とする。

ただし、実装・実機テスト時にmacOSが権限を要求する挙動が出ないか確認する。

通知機能を追加する場合のみ、通知許可が必要になる可能性がある。

## NF-007: 互換性

初期対象は以下。

```text
macOS 13 Ventura 以降
Apple Silicon Mac 優先
Intel Mac でも可能な範囲で対応
```

理由は、SwiftUIのメニューバーアプリ実装をシンプルにするため。

## NF-008: 保守性

アプリ内部は以下の責務に分離する。

```text
UI層:
メニューバー表示、メニュー項目、チェックマーク

Audio層:
入力デバイス取得、現在デバイス取得、切り替え

State層:
現在の一覧、現在選択中、エラー状態

Settings層:
ログイン時起動、表示設定

Infrastructure層:
Core Audio API、ServiceManagement API
```

## 12. 制約

Zoom、Google Meet、Discord、OBSなどのアプリ側でマイクが個別指定されている場合、MicBarでmacOSのデフォルト入力を変更しても、そのアプリに反映されないことがある。

MicBarはあくまで **macOSのデフォルト入力デバイス** を切り替えるアプリであり、各アプリ内部のマイク設定までは変更しない。

## 13. リスク

## R-001: デバイス名の重複

同名デバイスが複数あると、表示名だけでは区別しづらい。

## R-002: Bluetoothデバイスの不安定さ

AirPodsなどのBluetoothデバイスは接続・切断タイミングで一時的に一覧取得に失敗する可能性がある。

対策として、手動再読み込みとエラー耐性を用意する。

## R-003: 仮想オーディオデバイス

仮想デバイスによっては、入力デバイスとして表示されても実際には期待どおり使えない可能性がある。

対策として、初期バージョンではmacOSが入力デバイスとして返すものをそのまま表示する。

## R-004: 権限・Sandbox

App Store配布を考える場合、App Sandboxや権限周りの検証が必要になる。

初期バージョンでは個人利用・Developer ID配布を前提にする。

# `tech-stack.md`

# MicBar 技術スタック定義書

## 1. 技術選定の結論

MicBarは、**Swift + SwiftUI + Core Audio** で実装する。

UIはmacOSのメニューバー常駐アプリとして実装し、入力デバイスの取得・現在入力の取得・デフォルト入力の切り替えはCore Audioを使う。

SwiftUIにはメニューバー常駐UIを作るための `MenuBarExtra` があり、Appleは「アプリがアクティブでないときにもよく使う機能へアクセスさせる」用途として説明しているため、MicBarのような常駐ユーティリティと相性がよい。([Apple Developer][1])

## 2. 推奨スタック一覧

| 領域       | 採用技術                               | 用途                    |
| -------- | ---------------------------------- | --------------------- |
| 言語       | Swift                              | macOSネイティブアプリ開発       |
| UI       | SwiftUI                            | メニューバーのメニューUI         |
| メニューバー常駐 | `MenuBarExtra`                     | メニューバーアイコンとメニュー表示     |
| 音声デバイス制御 | Core Audio                         | 入力デバイス取得、現在入力取得、切り替え  |
| 状態管理     | `ObservableObject` / `@Published`  | デバイス一覧、現在入力、エラー状態     |
| 設定保存     | `UserDefaults`                     | ログイン時起動設定、表示設定        |
| ログイン時起動  | ServiceManagement / `SMAppService` | macOSログイン時の自動起動       |
| 通知       | UserNotifications                  | 将来的な切り替え通知            |
| 依存管理     | Swift Package Manager              | 外部ライブラリ追加時            |
| テスト      | XCTest                             | Audio層、State層のユニットテスト |
| 配布       | Developer ID + Notarization        | 自分以外へ配る場合             |
| CI       | GitHub Actions                     | 将来的な自動ビルド             |

Swift Package ManagerはSwift / Objective-C / C / C++などの再利用可能なコード部品を扱える仕組みとしてAppleが案内しているため、外部ライブラリを使う場合の標準的な依存管理にする。([Apple Developer][2])

## 3. 対象環境

## 3.1 macOS

推奨ターゲット。

```text
macOS 13 Ventura 以降
```

理由。

* `MenuBarExtra` を使ったSwiftUI実装に寄せられる
* `SMAppService` によるログイン時起動設定を使いやすい
* 古いmacOS対応でAppKit実装に寄せる必要がなくなる

`SMAppService` はmacOS 13以降でLogin Items、LaunchAgents、LaunchDaemonsの登録・制御に使うAPIとしてAppleが案内している。([Apple Developer][3])

## 3.2 CPU

対応方針。

```text
Apple Silicon: 優先対応
Intel Mac: 可能ならUniversal Binaryで対応
```

初期開発はBOSSのMac mini環境を主対象にする。

## 4. 実装方式

## 4.1 本命方式: Native Core Audio

最終的には、外部CLIに依存せず、Core Audioを直接呼ぶ。

Core Audioには、デフォルト入力デバイスを表す `kAudioHardwarePropertyDefaultInputDevice` があり、プロパティ取得には `AudioObjectGetPropertyData`、プロパティ変更には `AudioObjectSetPropertyData` を使う構成にする。([Apple Developer][4])

採用理由。

* Homebrew依存が不要
* `switchaudio-osx` のインストール不要
* 配布しやすい
* アプリ単体で完結する
* 将来的にログイン時起動や設定画面と統合しやすい

## 4.2 検証用方式: `switchaudio-osx`

プロトタイプ段階では、`switchaudio-osx` を呼び出して動作確認してもよい。

`switchaudio-osx` はMacの音声ソースをCLIから切り替えるツールで、`-t input` により入力デバイスを対象にでき、`-a` で一覧、`-c` で現在デバイス、`-s` でデバイス名指定、`-n` で次デバイスへの切り替えができる。([GitHub][5])

ただし、完成版ではランタイム依存にしない。

理由。

* Homebrew前提になる
* ユーザー環境によってパスが違う
* CLIの出力形式変更に影響される
* App Sandboxや配布時に扱いづらくなる
* 「アプリ単体で完結する」体験が弱くなる

## 6. Core Audio設計

## 6.1 主要な処理

Core Audio層では以下を実装する。

```text
1. すべてのオーディオデバイスを取得
2. 入力チャンネルを持つデバイスだけに絞る
3. デバイス名を取得
4. デバイスUIDを取得
5. 現在のデフォルト入力デバイスを取得
6. 指定デバイスをデフォルト入力に設定
7. デバイス一覧変更を監視
8. デフォルト入力変更を監視
```

## 6.2 主要API

| 用途        | API / Property                             |
| --------- | ------------------------------------------ |
| 全デバイス取得   | `kAudioHardwarePropertyDevices`            |
| 現在の入力取得   | `kAudioHardwarePropertyDefaultInputDevice` |
| 入力変更      | `AudioObjectSetPropertyData`               |
| プロパティ取得   | `AudioObjectGetPropertyData`               |
| デバイス名取得   | `kAudioObjectPropertyName`                 |
| デバイスUID取得 | `kAudioDevicePropertyDeviceUID`            |
| 入力対応判定    | `kAudioDevicePropertyStreamConfiguration`  |
| 変更監視      | `AudioObjectAddPropertyListenerBlock`      |

`AudioObjectAddPropertyListenerBlock` は、指定したAudioObjectのプロパティ変更通知を受けるためのAPIとしてAppleが提供している。([Apple Developer][6])

## 6.3 デバイスモデル

```swift
struct AudioInputDevice: Identifiable, Hashable {
    let id: AudioDeviceID
    let uid: String
    let name: String
    let isDefault: Bool
}
```

## 6.4 識別方針

UI表示には `name` を使う。

内部識別には `uid` を優先する。

一時的な切り替え処理には `AudioDeviceID` を使う。

理由。

* デバイス名は重複する可能性がある
* デバイスIDは再接続で変わる可能性がある
* UIDのほうが保存・再識別に向いている

## 7. UI技術

## 7.1 SwiftUI

UIはSwiftUIで実装する。

理由。

* メニュー構成が小さい
* 状態駆動で書きやすい
* `MenuBarExtra` と相性がよい
* 設定画面を追加する場合も自然に拡張できる

## 7.2 MenuBarExtra

基本構成。

```swift
@main
struct MicBarApp: App {
    var body: some Scene {
        MenuBarExtra("MicBar", systemImage: "mic") {
            MenuBarView()
        }
        .menuBarExtraStyle(.menu)
    }
}
```

`MenuBarExtra` はSwiftUI側でメニューバーExtraを作るための仕組みで、古いAppKit中心の実装より初期開発がシンプルになる。([Apple Developer][1])

## 7.3 AppKit fallback

macOS 12以前にも対応したくなった場合は、`NSStatusItem` に切り替える。

AppleのAppKitには `NSStatusItem` があり、`NSStatusBar` の `statusItem(withLength:)` でメニューバーに追加できる。([Apple Developer][7])

ただし、初期バージョンでは `MenuBarExtra` を優先し、AppKit fallbackは実装しない。

## 9. 設定保存

設定保存には `UserDefaults` を使う。

Appleは `UserDefaults` を、アプリ固有・システム全体の設定を保持するデフォルトシステムへのアクセス手段として説明している。([Apple Developer][8])

初期保存項目。

```text
launchAtLogin: Bool
showDeviceNameInMenuBar: Bool
lastSelectedDeviceUID: String?
```

初期バージョンで必須なのは `launchAtLogin` のみ。

`lastSelectedDeviceUID` は、将来的に「前回のマイクへ戻す」機能で使える。

## 10. ログイン時起動

ログイン時起動には `ServiceManagement` の `SMAppService` を使う。

設計。

```text
LoginItemService
├─ isEnabled()
├─ enable()
└─ disable()
```

macOS 13以降では、`SMAppService` がログイン項目などの登録・制御に使えるため、MicBarの対象OSと合っている。([Apple Developer][3])

## 11. 通知

初期バージョンでは通知はオフ。

将来的に通知を入れる場合は `UserNotifications` を使う。

AppleのUser Notifications frameworkは、ローカル通知をアプリ側で作成し、時間や条件に応じて配信する仕組みを提供している。([Apple Developer][9])

通知を入れる場合の方針。

```text
デフォルト: オフ
ユーザーが明示的にオンにした場合のみ通知許可を求める
通知は成功時よりも失敗時を優先
```

## 12. 権限とSandbox

## 12.1 マイク権限

MicBarは音声録音をしない。

MicBarはマイク入力ストリームを開かない。

MicBarはデフォルト入力デバイスの設定変更のみを行う。

したがって、初期設計ではマイク権限を要求しない。

ただし、これは実機で必ず検証する。

## 12.2 App Sandbox

初期バージョンでは、個人利用またはDeveloper ID配布を優先し、App Store配布は後回しにする。

App Sandboxは、アプリのアクセスをEntitlementにより制限してシステムリソースやユーザーデータを保護する仕組みであるため、App Store配布を考える段階ではCore Audio操作との相性を検証する。([Apple Developer][10])

## 13. 配布方針

## 13.1 個人利用

```text

↓

```

## 13.2 他人へ配布

他人へ配布する場合は、Developer ID署名とNotarizationを行う。

Appleは、Mac App Store外で配布するアプリについて、Developer ID署名によりGatekeeperが確認できるようにすること、さらにNotarizationで信頼性を高められることを案内している。([Apple Developer][11])

macOS 10.15以降では、Developer IDで配布するソフトウェアはNotarizationが必要になるため、公開配布時はNotarizationまで含める。([Apple Developer][12])

## 17. 採用しない技術

## 17.1 Electron

採用しない。

理由。

* 常駐アプリとして重い
* Core Audio操作に結局ネイティブブリッジが必要
* 小さなユーティリティには過剰

## 17.2 Python常駐スクリプト

採用しない。

理由。

* メニューバーUIとの相性が弱い
* 配布しにくい
* macOS権限や署名周りが面倒

## 17.3 AppleScript単体

採用しない。

理由。

* UIが弱い
* デバイス一覧取得や状態管理に向かない
* 長期的に保守しづらい

## 17.4 `switchaudio-osx`依存の完成版

完成版では採用しない。

理由。

* Homebrew依存
* CLIパス依存
* 配布体験が悪い
* アプリ単体で完結しない

ただし、プロトタイプや比較検証には使ってよい。

## 19. 開発方針まとめ

MicBarは、まず **入力マイク切り替えだけに集中する**。

初期バージョンでやること。

```text
メニューバーに置く
入力デバイスを表示する
現在の入力にチェックを付ける
クリックで切り替える
終了できる
```

初期バージョンでやらないこと。

```text
出力切り替え
音量調整
ミュート
録音
音声解析
AI機能
複数ショートカット
アプリ別自動化
```
